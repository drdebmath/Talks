Costs of Running Agents: The $20 Overnight Lesson

January 31, 2026: @BenjaminDEKR

A $20 Anthropic balance was drained overnight while OpenClaw kept checking whether it was daytime yet before sending a simple reminder ("get milk tomorrow").

The reported default heartbeat cadence was every 30 minutes, repeatedly pushing large context into an expensive model for a trivial decision.

x.com/BenjaminDEKR/status/2017644773356548532

Why It Went Viral

Approximate breakdown shared in-thread: about $0.75 per check, roughly 25+ checks, about $18.75 spent by heartbeat checks alone.

Core failure mode: no real sleep/wait primitive, no cost-aware model routing, and no hard overnight spend cap.

Community response: cheaper heartbeat models, longer intervals (for example 4 hours), local checks, and strict budget kill-switches.

Why Agents Magnify Tokens

Autonomous loops create repeated context re-sends: heartbeat checks, retries, tool logs, and long-memory prompts.

When each "small" check runs on premium models, token burn scales nonlinearly with runtime and task depth.

Practical controls: summarize context aggressively, route heartbeats to cheap models, cap context windows, and enforce per-loop token budgets.

OpenClaw: Structure + Function

System Structure (Control Plane)

Operational Function

  • Runs as a local-first personal assistant on your own device, usually installed with openclaw onboard --install-daemon.
  • Gateway is the control plane: it normalizes channel traffic, applies session rules, and dispatches work to the agent runtime.
  • The runtime executes tool actions (browser, canvas, cron, node actions), then reports results back to the source channel.
  • Supports always-on voice/talk surfaces and device-local capabilities through paired macOS/iOS/Android nodes.
  • Model layer supports OAuth/API keys, routing, and failover across provider choices or local model backends.
  • Security defaults treat inbound DMs as untrusted, using pairing and allowlists; stricter non-main sandboxing is available.

Designing an Agent: Functioning Loop

Why This Design Exists

  • Stateless chat forgets user context and tasks.
  • Passive UX cannot initiate useful work by itself.
  • No tools means no real-world execution.
  • Single channel fragments memory across apps.
  • Unsafe execution needs strong runtime guardrails.

Progressive Build (First Principles)

Start with one message handler. Real agency appears after four core loops are present.

State

Persist sessions and reload every turn.

Identity

Inject SOUL.md for stable behavior.

Agency

Tool loop: call, act, feed results, repeat.

Safety

Apply approval and allowlist policies.

Then scale into an always-on assistant

  • Gateway: unify Telegram/Discord/Slack/WhatsApp with shared identity.
  • Compaction + memory: keep context bounded while preserving long-term facts.
  • Queue + locks: prevent race conditions per session.
  • Heartbeats + multi-agent: proactive tasks and specialist delegation.
State Identity Agency Safety

Pattern summary: Sessions -> SOUL -> Tools -> Permissions -> Gateway -> Compaction -> Memory -> Queue -> Heartbeats -> Multi-agent.

From Message To Action To Memory

Context: From Chatbot Limits To Agent Design

  • Stateless chat becomes persistent by storing append-only JSONL sessions and compacting old context.
  • Generic behavior becomes identity-driven by injecting SOUL.md every turn.
  • Text-only replies become actions through structured tools and a repeated tool-use loop.
  • Unsafe execution risk is constrained with allowlists, approvals, and permission policy tiers.
  • Single-channel bots become unified assistants via a gateway pattern across all messaging surfaces.
  • One-size-fits-all agenting scales with multi-agent routing and shared long-term memory.
  • Passive usage becomes proactive operations through scheduled heartbeats and isolated cron sessions.
